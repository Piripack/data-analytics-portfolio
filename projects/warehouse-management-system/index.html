Database Systems Project

Warehouse Management System with SQLite and Tkinter

Designed and implemented a warehouse database system for inventory, orders, suppliers, employees, and storage locations using SQLite and Python. The project included a relational schema, Tkinter login and employee-management interfaces, XML export, BLOB image storage, and threaded order insertion.

SQLite Python Tkinter XML BLOB Storage Threading 3NF Design

Overview

This project was built around a warehouse operations scenario and focused on creating a structured database-backed system rather than only writing SQL queries in isolation. The implementation combined schema design, basic operational tooling, and data-handling features across the application layer.

What Was Built

  • A relational SQLite database covering employees, orders, products, suppliers, warehouse locations, login details, stock, and employee photos.
  • A Tkinter login workflow with SHA-256 password hashing and credential validation against the database.
  • An employee management interface for adding staff records through a GUI.
  • An XML export module that extracted order records into a structured orders.xml file.
  • BLOB storage and retrieval for employee photos inside the database.
  • A multi-threaded order insertion script that split order inserts into two concurrent batches.

Technical Evidence

  • The included database contains 8 non-system tables: EmployeePhotos, Employees, LoginDetails, Orders, Products, Stock, Suppliers, and WarehouseLocations.
  • The database currently holds 10 employees, 13 orders, 5 products, 5 suppliers, 5 warehouse locations, 1 login record, and 1 stored employee photo.
  • The codebase includes separate modules for GUI login, employee insertion, password hashing, XML export, BLOB retrieval, and threaded inserts.
  • The schema design and ERD clearly reflect a warehouse workflow with entity separation across employees, products, suppliers, orders, and locations.

Business Value

The system was designed to support common warehouse needs: stock visibility, supplier tracking, employee assignment, and order processing. The strongest portfolio value is that it shows a complete small system rather than a single script: database design, application logic, and data exchange features were combined in one deliverable.

Portfolio Summary

This project is strong portfolio material because it moves beyond database theory. It shows schema design, a real local database file, GUI-based interaction, hashed login validation, XML export, image storage, and concurrent processing in Python. For a student project, that breadth is materially useful.

One caution from the implementation review: the trigger definitions present in the database appear less reliable than the rest of the system because they reference columns or tables that are not clearly present in the current schema. For portfolio use, the strongest focus should stay on the schema, GUI, XML, BLOB, and threading work.
