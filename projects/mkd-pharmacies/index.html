MKD Pharmacies

A pharmacy ecommerce build that combines customer browsing, admin stock management, guarded checkout logic, and branch-level location browsing in one system.

Next.js App Router React Firebase Auth Firestore Tailwind CSS Leaflet Server Actions

Overview

MKD Pharmacies was built around a straightforward problem: let customers browse products and place orders, while giving staff enough control to manage stock and branch-level information. The public version of the project focuses on the parts that matter most on a portfolio page: user flow, admin flow, transaction logic, and system structure.

Under the surface, the app separates search and order handling, user state, admin tools, and map behaviour into distinct areas. That matters. It keeps the project from feeling like a single-page mock-up and makes the checkout and stock logic easier to reason about.

What Was Built

  • Product browsing with category filtering, keyword search, and add-to-basket interactions.
  • A basket and checkout flow with Firestore transaction logic to prevent overselling stock.
  • Firebase-backed sign-in and role-aware navigation for regular users and admins.
  • An admin inventory interface for creating and managing branch-specific products.
  • A branch locator page with a pharmacy map and location-aware stock browsing flow.

Technical Evidence

  • `src/actions/orders.js` uses `runTransaction` to validate current stock, update inventory, and create orders atomically.
  • `src/actions/search.js` keeps product search on the server and centralises that logic close to the data layer.
  • `src/context/CartContext.js` persists the basket in local storage and clears it when the user logs out.
  • `src/components/map/MapWrapper.js` dynamically loads the map with `ssr: false` to avoid server-side rendering conflicts.
  • `src/components/admin/AddProductForm.js` shows branch-aware product creation with dosage, expiry, usage, and side-effect fields.

Design and Architecture Value

  • The design report mapped nine use cases across search, ordering, authentication, admin stock control, order history, cancellation, and branch discovery.
  • The transaction sequence diagram adds genuine systems thinking by showing how concurrent checkout should fail safely when stock runs out.
  • The Firestore schema links users, products, pharmacies, and orders into a clear operational data model.
  • Together, those artefacts show the jump from planning into an implemented build with a clear operational shape.

Operational Value

  • The most important decision in this build was to treat stock accuracy as a control problem. That is why checkout uses transaction logic and server-side checks to protect inventory state.
  • The branch-level flow matters because it separates central ordering from local availability and gives the system a clearer operational model.
  • For healthcare or pharmacy-adjacent digital roles, the useful signal is the emphasis on access control, stock visibility, and failure-safe behaviour.

Scope and Notes

This works best as a portfolio project because it shows both structure and delivery. There is enough on the page to understand what was built, how stock safety was handled, and how the user and admin journeys differ.

One report claim about raw Leaflet usage was left out here because the implementation evidence showed `react-leaflet`. The page sticks to what could be checked cleanly.
